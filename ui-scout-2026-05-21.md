## 1. Shader.se — Scroll-Driven WebGPU Pipeline (Filip Kantedal case study) — https://tympanus.net/codrops/2026/05/19/80s-business-tech-seamless-scene-transitions-inside-shader-ses-scroll-driven-webgpu-pipeline/

WHAT IT DOES: A full-page interactive site whose scrolling DOM is fused with a Three.js/WebGPU scene graph rendered through a single post-processing pipeline. Distinct "scenes" (camera + lighting + meshes) live as React Three Fiber sub-trees, and a scroll position drives a TSL (Three Shading Language) crossfade node that linearly mixes the render targets of two adjacent scenes through a shared post stack (bloom, chromatic aberration, film grain). UI text and buttons are NOT separate DOM — they are rendered inside the canvas via `@pmndrs/uikit`, so headings inherit the same post effects (anti-aliasing pass, gradient noise) as the 3D geometry.

PACKAGES:
- next@15
- react@19
- three@0.170.x
- @react-three/fiber@9
- @react-three/drei@9
- @react-three/postprocessing@3
- three/tsl (built into three ≥ 0.170)
- @pmndrs/uikit@0.8.x
- @pmndrs/uikit-default
- leva (for in-dev shader knobs; strip in prod)
- gsap@3.13 + ScrollTrigger (for the scroll progress driver feeding TSL uniforms)
- maath (easing utilities for camera lerp)
- tailwindcss@4.0
- geist + jetbrains-mono (typography)

IMPLEMENTATION: Component tree is `<App>` → `<ScrollProvider>` (publishes `progress: MotionValue<0..N>`) → `<CanvasWebGPU>` (use `<Canvas gl={{forceWebGPU: true}}>` from R3F v9 with WebGL fallback) → `<PostStack>` (TSL-based: `bloom(mrt.emissive, 1.2)`, `dof(focusDistance, 0.02)`, `output = mix(sceneA, sceneB, smoothstep(scrollA, scrollB, scrollProgress))`) → `<SceneA/B/C>` each containing camera, lights, meshes, and a `<Root>` from `@pmndrs/uikit` for headings/CTAs. Animation timings: scene-to-scene crossfade is a 120vh scroll segment using `smoothstep` (do not use linear — banding shows on the bloom). Camera dolly per scene is `easeInOutCubic`, 1.4s on initial mount, then scroll-driven. Use `power3.out` ease for GSAP ScrollTrigger scrubbing with `scrub: 0.6` (smoothing factor) — never `scrub: true` (jitters at high refresh). Responsive: at <768px collapse three scenes into vertical-stacked `<Canvas>` instances (one per viewport-snap section) and disable bloom + DOF — keep only crossfade. Performance: cap DPR at `[1, 1.5]`, use `frameloop="demand"` outside scroll, drei `<PerformanceMonitor>` triggers DPR step-down at <55fps. UI text inside uikit: weight 500, tracking -0.02em, color `#E6E6E6` on `#0A0A0B`. Background grain: TSL `noise3D(uv * 800.0 + time*0.02) * 0.04` mixed in screen space — NOT a CSS overlay.

WHY IT WORKS: Hierarchy fusion — by rendering UI text inside the same post pipeline as the geometry, every glyph carries the same chromatic aberration and grain as the 3D meshes, so the eye reads "one surface" instead of "DOM-on-top-of-canvas." This is the Gestalt principle of common fate applied to visual treatment, not motion. The TSL crossfade replaces the usual abrupt scene-swap with a continuous lerp, which respects Hick's law: the user never has to re-orient because nothing teleports.

GOOD / BAD:
- GOOD: WebGPU-first with WebGL fallback is forward-looking; uikit-in-canvas eliminates the "two-rendering-systems" seam that ruins most R3F sites.
- BAD: Real SSR/SEO is gone — uikit text is canvas-rendered so crawlers see an empty `<canvas>`; requires a hidden semantic DOM mirror or a pre-rendered text-only route.

CLEAN APPLICATION: Apply this to Clean's landing-page hero by replacing the static "context layer for agents" headline with a TSL-crossfaded three-scene narrative — Scene 1: raw MCP traffic (particle storm, dense, noisy), Scene 2: TOON-compressed packet (geometry resolves into a clean lattice), Scene 3: agent reading from Clean (lattice flows into a single rendered prompt). Tie scroll progress 0→1 across hero (100vh) to the TSL crossfade `uMix` uniform. Render the H1, sub, and CTA via `@pmndrs/uikit` so they pick up the same film grain — this is the technical-credibility signal Clean needs.

## 2. aino.agency — https://aino.agency (Awwwards SOTD — May 20, 2026)

WHAT IT DOES: A monospace-driven editorial site for a Scandinavian design+technology agency where the entire layout is constructed in a fixed-width text grid. Hero is a Conway's-Game-of-Life-style cellular animation rendered in ASCII characters that respond to mouse position (character density modulates around the cursor with a falloff). All images are loaded then real-time rasterized to ASCII via canvas + luminance sampling; hovering an image toggles between raster and ASCII representations. Layout is a 12-col grid where every cell snaps to character width (`ch` units), creating perfect vertical alignment of body copy with image asci-edges.

PACKAGES:
- vite@5 + vanilla TS (no React — the site ships <50KB JS)
- gsap@3.13 (ScrollTrigger, Flip)
- lenis@1.1 (smooth scroll)
- p5@1.10 OR custom WebGL — they use a small custom `<canvas>` 2D + `requestAnimationFrame` loop, not p5
- fontsource-jetbrains-mono (single typeface)
- splitting@1.1 (per-character DOM splitting for hover effects)
- @vercel/og (for OG card generation that matches the ASCII aesthetic)

IMPLEMENTATION: Root layout `body { font-family: "JetBrains Mono"; font-size: 14px; line-height: 1.4; }` with CSS grid `grid-template-columns: repeat(80, 1ch);` — every cell is exactly one character wide. ASCII rasterizer: load image to offscreen canvas at 1/4 size, sample luminance per pixel, map to a 10-char ramp `" .:-=+*#%@"`, write to a contenteditable `<pre>` with `white-space: pre`. Mouse interaction: `pointermove` over the `<pre>` calculates `(charX, charY)`, applies a 12-character radius falloff that biases the ramp index up (brighter) within the radius — uses `gsap.quickTo` for sub-frame smoothing, easing `power2.out`, duration `0.3s`. Color: pure `#161615` background, `#F5F5F0` foreground, no gradients, no shadows. Section transitions: GSAP `Flip.from(state)` for layout shifts between scroll states (no opacity fades — the layout literally rearranges character cells). Type scale: 14/22/40/72px, all on a 4px baseline. Mobile: drop to 60-col grid at <640px, disable the cursor-driven density modulation (battery-prohibitive), keep the static ASCII art.

WHY IT WORKS: Monospace tells the engineer brain "this is a real tool, not marketing fluff" — the same psychological lever VS Code, Linear's changelog, and Vercel's dashboards pull. The ASCII rasterization is a single, consistent metaphor applied everywhere (images, dividers, even charts), giving an extraordinarily high coherence score: every element on screen obeys the same character grid, which is textbook Gestalt uniform connectedness.

GOOD / BAD:
- GOOD: Ships <50KB JS for a Site of the Day. The grid system is so internally consistent that the site reads as "system design," not "graphic design."
- BAD: Screen readers traverse the ASCII `<pre>` blocks as gibberish — needs aria-hidden on the visual `<pre>` and a parallel semantic block; current implementation does not.

CLEAN APPLICATION: Apply this to Clean's docs section by rendering the docs sidebar nav as a monospace tree (`├── installation`, `│   └── nextjs`, `└── api-reference/`) on a 1ch grid, with hover triggering a 12-char-radius luminance boost on the relevant entry. Match Clean's brand: `#0A0A0B` bg, `#E6E6E6` fg, accent `#7CFF6B` only on the active item. Use the ASCII rasterizer for the docs hero only (one large product diagram converted live) — leave the rest as readable text. This is the "Clean is for engineers" signal in one design move.

## 3. Capitolium — https://www.collabcapitolium.fr (Awwwards SOTD — May 18, 2026)

WHAT IT DOES: A campaign microsite for a Toulouse rugby+football jersey collab that uses a horizontal-scroll spine for the main narrative and a top navigation bar that "deploys" (vertically expands from a 56px pill into a full-bleed 280px overlay) as the user scrolls past the hero. Each "page" of the horizontal scroll is a full-bleed photographic plate where typography is locked to the foreground subject via CSS `clip-path` masks, creating the illusion that headings are physically printed on the jersey fabric.

PACKAGES:
- nuxt@3.13
- vue@3.5
- typescript@5.5
- gsap@3.13 + ScrollTrigger + Observer (for the wheel→horizontal redirection)
- lenis@1.1 (vertical) + custom horizontal proxy
- @nuxt/image (Cloudinary backend, AVIF + responsive `srcset`)
- splitting@1.1 (line + word splits for entrance)
- vite-plugin-glsl (for the subtle paper-grain shader)

IMPLEMENTATION: Layout shell: a vertical viewport that becomes a horizontal track at scroll-y > 100vh. Use GSAP `ScrollTrigger.create({ trigger: '#track', start: 'top top', end: '+=400%', pin: true, scrub: 0.8 })` with a tween animating `x: -75vw * (panels - 1)`. Wheel/touch handler uses `Observer.create({ type: 'wheel,touch,pointer', wheelSpeed: -1 })` to translate vertical wheel to horizontal scroll progress. Navigation deploy: navbar height tweens `56px → 280px` on `ScrollTrigger` with `start: 'top -10%'`, `toggleActions: 'play none none reverse'`, ease `expo.inOut`, duration `0.6s`. Inside the deployed nav, links stagger in with `gsap.from(links, { y: 24, opacity: 0, stagger: 0.04, ease: 'power3.out', duration: 0.5 })`. Type-on-jersey clip-path: foreground subject is a transparent-bg PNG with a hand-drawn SVG mask path; the H1 sits behind the PNG but in front of the background photo, with `clip-path: url(#jersey-mask)` cutting the text so it appears printed. Color palette `#F0E3CF` (warm cream) + `#58595B` (slate). Responsive: <1024px abandons horizontal scroll entirely — sections stack vertically and the deployed nav becomes a fullscreen overlay menu. WPO note: jury flagged WPO 6.80, so preload only the first panel's hero image, lazy-decode the rest with `loading="lazy" decoding="async"`.

WHY IT WORKS: The horizontal spine creates a forced rhythm — every panel takes the same scroll-distance to traverse, which is the temporal version of vertical baseline alignment. The deploy-on-scroll nav exploits progressive disclosure: the user only "earns" the full menu after committing to scroll past the hero, which keeps the initial hero composition uncontested by chrome.

GOOD / BAD:
- GOOD: The clip-path-on-jersey trick is a one-of-one move — no library would have generated it; it reads as craft.
- BAD: Accessibility scored 6.20 — horizontal scroll lockout with keyboard nav requires explicit arrow-key handlers, and the jury called it out. Don't ship horizontal scroll without `tab`+`arrow` parity.

CLEAN APPLICATION: Apply this to Clean's pricing page by making the three tiers (Hobby / Team / Enterprise) a horizontal-scroll spine where each tier is a full-bleed panel with a single screenshot of the dashboard variant for that tier (request counts, seat list, audit log depth). Deploy the top nav at scroll-y > 80% of hero to reveal docs + API reference + status as the user commits to evaluating tiers. Keep vertical-stack fallback under 1024px — no exceptions. Use Clean's existing JetBrains Mono for prices, Inter for body.

## 4. GSAP Gravity-Based Mouse Trail (Codrops tutorial, May 20, 2026) — https://tympanus.net/codrops/2026/05/20/made-with-gsap-building-a-fun-gravity-based-mouse-trail/

WHAT IT DOES: A canvas-less, pure-DOM cursor trail where each `pointermove` emits a thumbnail-sized image that spawns at the cursor, then is animated by a custom physics integrator: vertical gravity `9.8 * 60` px/s², horizontal velocity inherited from cursor delta, restitution `0.55` on viewport-bottom collision, friction `0.92` per second. Trails decay via opacity over 2.4s and are GC'd from the DOM after coming to rest. Total scene runs at 60fps with 80+ concurrent elements because GSAP's `quickSetter` writes transforms in a single rAF batch.

PACKAGES:
- gsap@3.13 (only core — no plugins required)
- vanilla TS or react@19
- vite@5
- no animation library beyond GSAP — the physics is hand-rolled

IMPLEMENTATION: On mount, create a pool of 120 `<img>` elements positioned `position: fixed; pointer-events: none; will-change: transform; top: 0; left: 0;`. Track `pointermove` with throttled `requestAnimationFrame` at most every 50ms emit. State per element: `{x, y, vx, vy, life}`. Per-frame loop (single `gsap.ticker.add`): `vy += 0.5; x += vx; y += vy; if (y > window.innerHeight - 60) { y = window.innerHeight - 60; vy *= -0.55; vx *= 0.7 }`. Use `gsap.quickSetter(el, 'css')` to batch-write `transform: translate3d(${x}px, ${y}px, 0) rotate(${rotation}deg)` — quickSetter is 5x faster than `gsap.set` in a tight loop. Rotation: `rotation += vx * 0.05` (spin proportional to horizontal velocity). Initial vx/vy on emit: `(cursor.x - last.x) * 0.6` and `(cursor.y - last.y) * 0.6 - 4` (slight upward kick). Lifetime: `gsap.to(el, { opacity: 0, duration: 0.6, delay: 1.8, onComplete: () => pool.return(el) })`. Cap pool — never let DOM elements exceed 120 or paint times spike.

WHY IT WORKS: Physical metaphors (gravity, bounce, friction) are pre-loaded in users' intuition — they don't have to "learn" the interaction. Combined with delight-by-surprise (the assets are real product screenshots / icons, not abstract dots), it creates a moment of "oh, this is a real engineering team" that converts at the hero better than any tagline.

GOOD / BAD:
- GOOD: Zero libraries beyond GSAP, runs at 60fps on a 2019 MacBook Air with 80 elements — the kind of perf result that proves the implementation is clean.
- BAD: Easily becomes obnoxious — if applied to a serious B2B page, the gimmick fights the content. Strict limits on what triggers emission (only over hero, only when cursor is moving fast).

CLEAN APPLICATION: Apply this to Clean's landing-page hero ONLY (not docs, not pricing) as a subtle "MCP packet" cursor trail — emit a tiny JSON-glyph badge (`{}`) every 80ms while the cursor is inside the hero bounding box, with gravity pulling them down toward a fixed "Clean" sink at the bottom-center of the hero. The sink absorbs them with a brief scale-down → poof. This visualizes the product metaphor (raw context falls into Clean) without any explicit annotation. Disable on `prefers-reduced-motion`.

## 5. Aceternity UI — Pixelated Canvas + Background Ripple Effect (May 2026 drops) — https://ui.aceternity.com/components/pixelated-canvas + https://ui.aceternity.com/components/background-ripple-effect

WHAT IT DOES: Pixelated Canvas converts a source image to a grid of sampled dots/squares on a `<canvas>`, then runs a per-pointer-position physics distortion (modes: `repel`, `attract`, `swirl`) within a configurable radius. Background Ripple is a grid of CSS-only cells where a click triggers a radial ripple of `transform: scale()` + `background-color` animations propagating outward from the click point using GSAP-like stagger. Both are install-as-a-block via shadcn registry — single `npx shadcn add` command.

PACKAGES:
- shadcn (CLI: `npx shadcn@latest add @aceternity/pixelated-canvas`)
- next@15 + react@19
- tailwindcss@4
- motion@11 (Framer Motion renamed; Aceternity's new components require this exact major version)
- clsx + tailwind-merge (shadcn utility deps)
- lucide-react (icons)

IMPLEMENTATION: Pixelated Canvas — `<PixelatedCanvas src="/hero.webp" width={1280} height={640} cellSize={4} dotScale={0.85} shape="square" backgroundColor="#0A0A0B" tintColor="#7CFF6B" tintStrength={0.15} distortionMode="swirl" distortionStrength={3} distortionRadius={80} followSpeed={0.2} jitterStrength={4} jitterSpeed={4} fadeOnLeave fadeSpeed={0.1} maxFps={60} />`. Critical knobs: keep `maxFps={60}` (the component ships with no cap by default and will burn CPU on retina); `dropoutStrength={0.4}` thins dots in low-contrast regions so dark backgrounds stay clean. Background Ripple — wrap any section in `<BackgroundRippleEffect rows={20} cols={32} cellSize={56} interactive />`; the ripple originates from `event.target` cell with `stagger: 0.012`, `duration: 0.6`, ease `cubic-bezier(.2,.8,.2,1)`. Both components are layout-aware via ResizeObserver — but Pixelated Canvas at `responsive={true}` re-samples the source image on every resize, which is expensive — debounce window resize to 200ms in a wrapper.

WHY IT WORKS: Both effects let static brand imagery feel "alive" with one prop. The Pixelated Canvas in particular makes the implicit promise "we have something to show, but it's resolution-locked behind interaction" — this is a curiosity gap, the strongest CTR driver in landing-page hierarchy. The square-cell grid also visually parses as "data," not "photo," which aligns with technical brand voice.

GOOD / BAD:
- GOOD: Copy-paste-ship in <5 minutes; props are exhaustive (28 documented).
- BAD: Both components use the Tailwind v4 + Motion 11 ecosystem; if your project is on Tailwind v3 or Framer Motion 10, the install will silently fail at runtime. Lock dependencies before pulling.

CLEAN APPLICATION: Apply Pixelated Canvas to Clean's landing-page hero as a converted-to-dots rendering of the Clean dashboard screenshot, with `distortionMode="repel"` and `tintColor="#7CFF6B"` — the dashboard reads as "data" not "screenshot," which is what dev-infra buyers want to see. Apply Background Ripple to the pricing-page CTA section only, with `cellSize={48}`, originating from the "Get API key" button on click — the ripple confirms the action in a way no button-state animation can.

## BUILD DIRECTIVE

TARGET: Clean's landing-page hero (tryclean.ai). One surface, one rebuild today.

APPROACH: Build a three-scene scroll-driven hero using React Three Fiber + WebGPU (pick #1) as the core architecture, but ship two tasteful overlays from picks #4 and #5 — a GSAP cursor-emission of `{}` JSON packets falling into a sink (visualizing the product metaphor) and a Pixelated Canvas dashboard preview placed in scene 3 so the resolved end-state of the narrative is the actual Clean dashboard rendered as data dots. KEY CONSTRAINTS: Clean is developer infrastructure — every visual decision must read as "engineering output," not "marketing campaign." Avoid: gradient meshes, glassmorphism cards, generic blob backgrounds, "AI sparkles," lavender/orange Linear-clone gradients, any Framer Motion preset that's recognizable as a preset, looping abstract videos. Cap total page weight at 250KB JS + 600KB media on initial paint. Bloom must be subtle (`luminanceThreshold: 0.9, intensity: 0.6` max) — anything more reads as gaming-site, not dev-tool. Color discipline: `#0A0A0B` background, `#E6E6E6` foreground, `#7CFF6B` is the only chromatic accent and only appears on the primary CTA and on active states. Typography: JetBrains Mono for any UI/code-feel surface, Geist Sans for narrative copy, nothing else. All interactive flourishes must be disabled under `prefers-reduced-motion`.
