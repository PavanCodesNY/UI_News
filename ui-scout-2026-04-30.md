## PieterKoopt® — https://www.awwwards.com/sites/pieterkoopt-r

WHAT IT DOES: Single-page developer portfolio that treats the viewport as a horizontally-paginated spatial grid driven by a fixed-pixel canvas under a CSS Grid overlay. A persistent WebGL layer (full-bleed `position: fixed`) renders chromatic-aberration text masks and noise distortion behind the DOM, while the foreground is built in pure semantic HTML so the page stays Lighthouse-clean. Scroll is hijacked through Lenis with a 1.2 multiplier, and section transitions happen via shared-element `view-transition-name` morphs between project tiles and their detail views — not via route changes.

PACKAGES:
- next@15 (App Router, RSC for shell, client islands for canvas)
- three@0.165 + @react-three/fiber@8.16 + @react-three/drei@9.108 + @react-three/postprocessing@2.16
- gsap@3.12 + ScrollTrigger + SplitText
- @studio-freight/lenis@1.1
- tailwindcss@3.4 with `@tailwindcss/typography`
- next/font with `geist-mono` (numeric tabular) + `inter-display` (headers)
- lucide-react@0.383 for icon set

IMPLEMENTATION:
1. Shell: `<html data-lenis-prevent>` with a `Lenis` provider mounted at root. `lerp: 0.08, duration: 1.2, smoothWheel: true`. Sync `ScrollTrigger.update()` to Lenis's `scroll` event in a `useGSAP()` hook.
2. WebGL stage: a single `<Canvas dpr={[1,2]} gl={{ antialias:false, powerPreference:'high-performance' }}>` mounted `fixed inset-0 -z-10 pointer-events-none`. Inside, a `<ScreenQuad>` runs a shader that samples a `RenderTarget` of an offscreen scene where each "section" is a `<Text>` mesh. A uniform `uPage` (0..N) is animated with `damp(uPage, target, 6, dt)`.
3. Layout: `grid-cols-12 gap-x-[clamp(8px,1vw,16px)] px-[clamp(16px,2vw,32px)]`. Baseline grid is exposed in dev with a key-toggle that overlays `bg-[linear-gradient(to_right,_transparent_calc(100%/12_-_1px),_rgba(255,255,255,0.06)_calc(100%/12))]`.
4. Type: H1 `font-size: clamp(72px, 11.2vw, 220px); letter-spacing: -0.045em; line-height: 0.86`. Body `15px/1.55`. Numerics use `font-variant-numeric: tabular-nums lining-nums`.
5. Animation primitives: every micro-interaction uses the same easing — `cubic-bezier(0.65, 0, 0.35, 1)` over `420ms` for "snap" and `cubic-bezier(0.2, 0.8, 0.2, 1)` over `900ms` for "settle." No bouncy springs.
6. View transitions: each project tile has `viewTransitionName: project-${slug}`. On click, the canvas triggers a 700ms displacement reveal while the tile morphs into the hero image of the detail page.
7. Performance: `IntersectionObserver` to pause the canvas's `frameloop` to `'demand'` when the tab is hidden. Use `<Preload all />` from drei. Defer all GLTFs behind `Suspense` with skeleton boxes.

WHY IT WORKS: Strict 12-column grid + monospaced numerics signals engineering rigor; the floating WebGL layer adds optical density without taxing DOM. Motion is consistent (one curve, one duration family), which the brain reads as "intentional" rather than "decorative." Negative space at 18–22% of the viewport keeps cognitive load low while letting display type carry the message.

GOOD / BAD:
- Good: Best-in-class hierarchy from a single typeface, locked-grid discipline, motion that never blocks reading.
- Bad: Hijacked scroll fights trackpad inertia on Windows; canvas costs ~120MB GPU on entry — punishing on 4GB Chromebooks.

CLEAN APPLICATION: Apply this to Clean's landing-page hero by making the headline a WebGL-rendered text mesh ("Context for every agent. Without the context window tax.") that reacts to cursor proximity with a 0.6-radius chromatic split, while the rest of the page stays plain DOM. The fixed canvas should also render a soft, slow noise-field as background through every section, never attached to the scroll position — so the tech feels ambient rather than gimmicky.

## Lusion (V3 + labs.lusion.co) — https://lusion.co

WHAT IT DOES: Studio site built as a single GPU scene with deferred-DOM overlays. The hero is a real-time particle field driven by an FBO ping-pong simulation (positions in one render target, velocities in another) with curl noise advection. Project cards are 3D planes in the same scene; hovering swaps a screen-space mask shader from a static thumbnail to a video texture with a feathered cursor-radius reveal. Navigation is a magnetic dock that snaps to a Bézier path along the scene.

PACKAGES:
- three@0.165 + @react-three/fiber + @react-three/drei + leva (debug)
- @react-three/rapier@1.4 (only on /labs subroutes, for collision demos)
- gsap@3.12 + Flip plugin
- maath@0.10 for buffer helpers
- glslify-loader for `.glsl` shader imports
- vite@5 build (not Next — Lusion ships Vite + React)
- ts-pattern for state machines on cursor states

IMPLEMENTATION:
1. Single canvas at root, full viewport, `gl={{ alpha: true, premultipliedAlpha: false, powerPreference: 'high-performance', antialias: false }}`, MSAA replaced with FXAA via postprocessing.
2. Particle FBO: two `WebGLRenderTarget(512, 512, { type: FloatType })`. Positions shader advects with `curlNoise(p * 0.4 + uTime*0.05) * 0.6`. Render 262,144 instances as a `Points` mesh; `gl_PointSize = (uSize * uPixelRatio) / -mvPosition.z`.
3. Project tiles: `<mesh>` planes laid out on a curved spline; `useScroll` from drei provides `scroll.offset`. A custom `ShaderMaterial` blends `uMapA → uMapB` via a feathered `smoothstep(uRadius - 0.1, uRadius + 0.1, length(uv - uMouse))`.
4. DOM overlay: project titles in DOM positioned with `useFrame` projecting world coords to screen — `vec3.project(camera)` → `style.transform = translate3d(...)`. Keeps copy SEO-indexable and accessible.
5. Cursor: custom 14px ring; on hover targets, scales to 64px with `mix-blend-mode: difference`. Spring config: `stiffness: 220, damping: 26, mass: 0.8`.
6. Page transitions: GSAP Flip captures DOM rectangles, then runs a `0.9s` ease `power3.inOut` morph while the canvas plays a synchronized "warp" shader (UV noise scale 0.0 → 1.4 → 0.0).
7. Performance: clamp `pixelRatio` to `Math.min(devicePixelRatio, 1.75)` on retina; on iOS, force `1.0`. Disable particles entirely under `prefers-reduced-motion`.

WHY IT WORKS: Particle density creates parallax depth without literal 3D objects, so the eye gets atmosphere without semantic noise. Mix-blend-mode cursor establishes immediate "this is interactive" feedback across every surface. Co-locating a single canvas under everything yields one paint per frame — the page feels weightless because there are no compositor reflows.

GOOD / BAD:
- Good: One render path for all motion is the technically correct architecture for a creative studio site; cursor system communicates affordance without copy.
- Bad: ~6MB shader+texture payload before first interaction; single-canvas pattern is brutal to debug without a Leva panel.

CLEAN APPLICATION: Apply this to Clean's product demo section by replacing the static "MCP proxy" diagram with a GPU particle field where each particle represents a token flowing through the proxy. Color particles by route: green for cached, purple for proxied, dim grey for stripped-out tokens. Cursor proximity reveals the underlying route metadata as a DOM tooltip. The visual literally is the product's value prop — context routing, made visible.

## Cursor 3 — https://cursor.com/blog/cursor-3

WHAT IT DOES: Marketing/launch page for Cursor 3 ("Glass") that demos the Agents Window through an in-page interactive video shell. Above-the-fold is a glassmorphic panel UI rebuild that responds to scroll: as you scroll, agent cards animate in horizontally with staggered translate+blur, and a scrubbable demo (looped MP4 with `currentTime` mapped to scroll progress) plays the multi-agent workflow. Typography is Inter Tight at heavy weights with a dense `letter-spacing: -0.04em`.

PACKAGES:
- next@15 (RSC + edge runtime)
- framer-motion@11
- tailwindcss@3.4 + tailwindcss-animate
- shadcn/ui (Card, Tabs, Dialog) on top of Radix
- @vercel/og for dynamic OG image rendering
- lucide-react
- next/font Inter Tight + JetBrains Mono
- @vidstack/react for accessible video shell (with `currentTime` exposed)

IMPLEMENTATION:
1. Hero: `min-h-[100svh]` (small viewport unit), `grid-rows-[auto_1fr_auto]`. Headline `text-[clamp(56px,7vw,128px)] tracking-[-0.04em] leading-[0.95]`. Sub `text-[clamp(16px,1.4vw,20px)] text-zinc-400 max-w-[58ch]`.
2. Background: layered (1) `radial-gradient(ellipse at 50% 0%, rgba(120,140,255,0.18), transparent 50%)`, (2) noise SVG at 4% opacity, (3) horizon line at 70vh with `box-shadow: 0 0 200px 40px rgba(160,180,255,0.25)`.
3. Glass panel: `backdrop-filter: blur(20px) saturate(140%); background: linear-gradient(180deg, rgba(255,255,255,0.06), rgba(255,255,255,0.02)); border: 1px solid rgba(255,255,255,0.08); box-shadow: inset 0 1px 0 rgba(255,255,255,0.08), 0 30px 80px -20px rgba(0,0,0,0.6)`.
4. Scroll-scrub video: pin section with `position: sticky; top: 0; height: 100vh`. Use `useScroll` + `useTransform` from framer-motion to map `scrollYProgress (0..1)` to `videoRef.currentTime = duration * progress`. Preload video as `metadata` then upgrade to `auto` once in viewport.
5. Agent cards: each card uses `motion.div` with `initial={{ y: 20, opacity: 0, filter: 'blur(8px)' }}`, `whileInView={{ y: 0, opacity: 1, filter: 'blur(0)' }}`, `transition={{ duration: 0.55, ease: [0.2, 0.8, 0.2, 1], delay: i * 0.06 }}`.
6. Code/terminal blocks: real `<pre>` with shiki-rendered tokens; cursor blink is a CSS `animation: blink 1s steps(2) infinite`. Shows actual diff rather than image.
7. Section rhythm: vertical spacing follows a 4-tier scale: `py-24` (default), `py-32` (transitions), `py-16` (compressed), `py-40` (set pieces). Never fall off the scale.

WHY IT WORKS: The page is the product. Showing a real diff and a real agent flow inside the marketing surface short-circuits skepticism — the reader's brain treats "interactive demo" as evidence rather than copy. Heavy negative tracking + mono code creates a tonal contract: serious tool, not a toy. Glass with strict inset highlight reads as "OS-native quality" without the cliché of frosted hero cards over photography.

GOOD / BAD:
- Good: Scroll-scrubbed real product video, type system locked to a single scale, copy that names the abstraction ("Agents Window") rather than the feature.
- Bad: Glass blur destroys battery on M1 Air at 4K; sticky-pin sections break Safari's anchor restore.

CLEAN APPLICATION: Apply this directly to Clean's landing page below the hero. Replace the standard "feature grid" with a scroll-scrubbed terminal recording showing a Claude agent making 18 MCP calls — and a side-panel that updates in real time to display: tokens-before, tokens-after-Clean, latency, dollars saved. The scroll-scrub of a real terminal recording is the most honest demo Clean can ship; it makes the value of the proxy concrete in the same way Cursor 3 makes the agents window concrete.

## Aceternity UI — Webcam Pixel Grid — https://ui.aceternity.com/components/webcam-pixel-grid

WHAT IT DOES: A drop-in React component that requests camera permission, streams the user's webcam to an offscreen `<canvas>`, downsamples each frame to a pixel grid (default 64×48), and renders the grid as a `<div>` matrix where each cell's brightness drives a CSS variable for color and scale. The result is a real-time "you are here" pixelated mirror that turns visitors into part of the marketing surface.

PACKAGES:
- react@18 + next@15 (client component only)
- tailwindcss@3.4
- framer-motion@11 (only for the permission prompt animation)
- No webgl required — pure 2D canvas + CSS transforms

IMPLEMENTATION:
1. Component shell: `'use client'`. `useRef` for hidden `<video>`, hidden `<canvas>`, and grid container.
2. Capture: `navigator.mediaDevices.getUserMedia({ video: { width: 640, height: 480, facingMode: 'user' } })`. Attach to `videoRef.current.srcObject`.
3. Sampling loop: in `requestAnimationFrame`, draw video onto offscreen canvas at `gridW × gridH` (e.g., 64×48). `getImageData(0,0,gridW,gridH).data` returns RGBA buffer. Throttle to 30fps with a `lastTs` check.
4. Grid render: build a `gridW * gridH` array of `<div>` once on mount. Update via direct DOM (`el.style.setProperty('--b', brightness)`) — never via React state per cell. Each cell: `width: calc(100% / 64); aspect-ratio: 1; background: rgb(var(--b),var(--b),var(--b)); transform: scale(var(--s));`.
5. Style modes: provide a `mode` prop — `'mono' | 'duotone' | 'invert' | 'ascii'`. ASCII mode replaces the cell with a single character chosen from `' .:-=+*#%@'[Math.floor(b/26)]`.
6. Privacy: a `requestCamera()` button triggers permission only on user click. Default state shows a neutral pixelated placeholder (a static seed image) so first paint isn't empty.
7. Accessibility: full `aria-hidden` on the grid; provide a `<button>` to toggle off and a visible "your camera is being used in your browser only" disclaimer line.

WHY IT WORKS: Self-presence triggers the highest engagement of any web pattern (mirror selfie effect, ~3× dwell time on Awwwards mirror demos). Pixelating it depersonalizes — feels playful rather than surveillant. CSS-variable-per-cell + direct DOM updates avoid React's reconciliation cost, sustaining 30fps on a 3072-cell grid on a base M1.

GOOD / BAD:
- Good: Zero-WebGL technique that nonetheless feels novel; great accessibility default with explicit opt-in; ships as a single `tsx` file under 200 lines.
- Bad: Surveillance optics — even with pixelation, asking for camera on a marketing site has a real bounce-rate cost; mobile Safari + iOS 17 occasionally drops the stream after backgrounding.

CLEAN APPLICATION: Do not put the literal webcam grid on Clean's marketing site — wrong vibe for developer infra. Instead, port the pattern to a "Live Token Grid": each cell renders one token from a real-time stream of context being processed by Clean (sourced from a sample MCP session log, replayed on a loop). Cell color encodes route: green for kept, amber for compressed, red for dropped. The grid breathes with traffic, and hovering a cell shows the actual token text in a tooltip. Same psychological hook as the webcam — "the page is alive and reflecting something real" — but speaking the developer's language.

## Ruinart — Digital Fresco — https://www.awwwards.com/sites/ruinart-digital-fresco

WHAT IT DOES: Immersive narrative scroll that combines a horizontally-scrolling SVG fresco (≈18,000px wide, layered into 5 parallax planes) with synchronized audio cues and DOM annotations that fade in along a polyline path. Built as a "scroll-driven painting" — vertical wheel input is mapped to horizontal travel through the scene with eased pinning. Uses `prefers-reduced-motion` to fall back to a stepper UI with prev/next buttons.

PACKAGES:
- nuxt@3 (Vue, but pattern translates 1:1 to Next/React)
- gsap@3.12 + ScrollTrigger + DrawSVGPlugin (paid) + MotionPath
- @studio-freight/lenis
- howler@2.2 for spatialized audio
- splitting@1.0 (or `splitType` in JS) for character-level reveals
- tailwindcss@3.4

IMPLEMENTATION:
1. Stage: `<section class="h-[500vh] relative">` with an inner `position: sticky; top: 0; height: 100vh; overflow: hidden;` containing the horizontal track.
2. Horizontal track: `<div class="track absolute inset-y-0 left-0 flex" style="width: 18000px">`. ScrollTrigger maps `scrollYProgress` to `gsap.set(track, { x: -progress * (18000 - viewportW) })`.
3. Parallax planes: 5 `<svg>` layers stacked, each translating at `0.4, 0.6, 1.0, 1.3, 1.6` of base velocity. Build `transform: translate3d(...)` with `will-change: transform` only during scroll (toggled on `scrollstart` / `scrollend`).
4. Path annotations: a hidden `<path>` defines the narrative spine. Use GSAP MotionPath to animate a `<g>` of label DOM elements along the path, with `start: progress, end: progress + 0.05` triggers per beat.
5. Audio: Howler instances per scene, `pos(x, y, z)` updated each frame against the player's "virtual position" along the track. Master gain ramps with `prefers-reduced-motion`.
6. Reduced motion: detect at mount, then replace the scroll system with a stepper — same scenes, same annotations, controls via Left/Right keys + on-screen prev/next.
7. Performance: SVGs >5MB are sliced into 5 separate files, loaded on demand by ScrollTrigger callbacks. Convert any large gradients to `<rect>` + `<linearGradient>` rather than embedded raster.

WHY IT WORKS: Linear horizontal narrative aligns with the way humans read a fresco IRL — the metaphor is the form. Audio + parallax + character reveals together create a "stepping into a museum" frame, which gives the brand permission to show off. Reduced-motion fallback isn't an afterthought; it's a real second design that respects the same content beats.

GOOD / BAD:
- Good: World-class craft on the parallax timing; audio is panned spatially, which is rare and well-done; reduced-motion alternative is a real design.
- Bad: 18,000px SVG murders Android Chrome with composite layers >2GB; on slow networks the first-paint is a 4-second blank canvas before any asset arrives.

CLEAN APPLICATION: Apply this to Clean's "How Clean works" docs/explainer page (replace the current vertical wall-of-text). Build a horizontal fresco of a single agent request's lifecycle: client → Clean proxy → MCP server → response → context layer → return. Each annotation reveals a snippet of the actual TOON-format payload at that step. Use SVG line art (technical drafting style, not illustrative) and a soft mono-purple palette so the fresco reads as engineering schematic rather than brand campaign.

## BUILD DIRECTIVE

TARGET: Clean's landing page hero + immediately-below-fold "live demo" section (single deliverable, two surfaces stitched together).

APPROACH: Build a hero whose headline is a WebGL-rendered text mesh with subtle chromatic split on cursor proximity (PieterKoopt pattern), backed by a single fixed Canvas that also hosts a sparse particle field representing tokens flowing through the Clean proxy (Lusion FBO pattern, but density-restrained — atmosphere not spectacle). Directly below the hero, pin a 100vh section with a scroll-scrubbed real terminal recording of a Claude agent making MCP calls (Cursor 3 pattern), and a side panel showing live numerics — tokens before/after Clean, latency, $ saved — that update synchronously with `scrollYProgress`. Finish the section with a "Live Token Grid" component (Aceternity Webcam Pixel Grid pattern, ported to token visualization) that loops a real captured MCP session and lets developers hover any token to see how Clean handled it. Reduced-motion fallback must be designed, not retrofitted (Ruinart pattern).

STACK:
- next@15 (App Router, RSC shell, client islands for canvas + scroll-scrub)
- three@0.165 + @react-three/fiber@8.16 + @react-three/drei@9.108 + @react-three/postprocessing@2.16
- framer-motion@11
- @studio-freight/lenis@1.1
- gsap@3.12 + ScrollTrigger
- tailwindcss@3.4 + tailwindcss-animate
- shadcn/ui primitives (Card, Dialog, Tabs)
- shiki for accurate code/diff rendering in the terminal recording overlay
- next/font: Inter Tight (display) + JetBrains Mono (numerics, code, terminal)
- lucide-react

REFERENCE URLS:
- https://cursor.com/blog/cursor-3 (scroll-scrub demo + glass panel + type system)
- https://lusion.co (single-canvas architecture + cursor system)
- https://www.awwwards.com/sites/pieterkoopt-r (grid discipline + WebGL text behind DOM)

KEY CONSTRAINTS: Clean is developer infrastructure — the result must read as engineering, not as an art-school portfolio. Specifically:
- No splash intros, no "scroll to enter," no preloader screens. First paint must be useful copy in <800ms.
- No bouncy springs, no playful confetti. Single easing family: `cubic-bezier(0.2, 0.8, 0.2, 1)` for settles, `cubic-bezier(0.65, 0, 0.35, 1)` for snaps. No `mass`-driven physics.
- No hero photography or stock illustration. All visual interest must come from type, grid, code, and the WebGL token field.
- No camera/microphone prompts. The pixel-grid pattern must be ported to token data, never to webcam.
- Performance budget: Lighthouse Performance ≥90 on a Moto G4 simulation. Total JS ≤180KB gzipped before the canvas chunk loads. Canvas chunk lazy-loaded behind `IntersectionObserver`, not on first paint.
- No glass blur stacked >2 layers deep. Backdrop-filter is taxed to one container per viewport.
- Mono numerics are mandatory wherever Clean shows a metric. `font-variant-numeric: tabular-nums` everywhere.
- Never claim "10x faster" or similar marketing puffery. All numbers must be real, sourced from a measurable demo session, with a footnote linking to the raw log.
